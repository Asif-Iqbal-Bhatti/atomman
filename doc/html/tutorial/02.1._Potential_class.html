1. IntroductionÂ¶

The atomman package is designed to interact with LAMMPS-compatible interatomic potential implementations and assist with the automatic and dynamic generation of the associated LAMMPS command lines. Starting with atomman version 1.3.3, the main management of LAMMPS potentials is now handled with the potentials package, with atomman inheriting classes for representing LAMMPS potentials and accessing database records from the NIST Interatomic Potentials Repository. These methods allow for pre-defined potentials to be retrieved either from the remote database potentials.nist.gov or from a local directory, and for user-defined potentials to be easily integrated in with the official ones.

2. Loading LAMMPS PotentialsÂ¶

2.1. Using load_lammps_potentialÂ¶

Added version 1.3.3

The function atomman.load_lammps_potential allows for a LAMMPS potential to be loaded from the potentials database. Depending on the settings used, this function will search for matching potentials from potentials.nist.gov and/or from the local copy of the database.

Search parameters

  • id (str or list, optional) The id value(s) to limit the search by.

  • key (str or list, optional) The key value(s) to limit the search by.

  • potid (str or list, optional) The potid value(s) to limit the search by.

  • potkey (str or list, optional) The potkey value(s) to limit the search by.

  • status (str or list, optional) The status value(s) to limit the search by.

  • pair_style (str or list, optional) The pair_style value(s) to limit the search by.

  • elements (str or list, optional) The included elemental model(s) to limit the search by.

  • symbols (str or list, optional) The included symbol model(s) to limit the search by.

Access options

  • verbose (bool, optional) If True, informative print statements will be used.

  • getfiles (bool, optional) If True, then the parameter files for the matching potentials will also be retrieved and copied to the working directory. If False (default) and the parameter files are in the library, then the returned objectsâ pot_dir path will be set appropriately.

Database options

  • database (atomman.library.Database, optional) A pre-existing Database class to use for accessing the records. Useful if multiple potentials are being retrieved as records can be pre-loaded for efficiency. If not given, a new database object will be initialized based on the parameters below.

  • localpath (str, optional) The path to the local database directory to use. If not given, will use the default potentials database path. Ignored if database is given.

  • local (bool, optional) Indicates if the local version of the library should be searched. Default value is True. Ignored if database is given.

  • remote (bool, optional) Indicates if the remote version of the library should be searched. Default value is True. Ignored if database is given.

Returns

  • (potentials.PotentialLAMMPS) The potential object to use.

2.1.1. By idÂ¶

Each LAMMPS potential in the NIST Interatomic Potentials Repository has a unique id that indicates the associated publication, what the model represents, and which archived version it is for. These idâs can be easily found by either searching for potentials using

  • Interatomic Potentials Repository and finding the ids for the LAMMPS versions you want to use - the idâs in parenthesis that contain âLAMMPSâ within them.

  • potentials.nist.gov and in the âFilter by Templateâ section on the left clicking the box next to potential_LAMMPS.

[2]:

potential = am.load_lammps_potential('1989--Adams-J-B--Ni--LAMMPS--ipr1')
print(potential)

2.1.2. By potidÂ¶

You can also search using the potential model id - the main identifier for potentials on the Interatomic Potentials Reporitory - using the potid parameter. For most but not all potentials, there is only one âactiveâ LAMMPS version, so searching in this manner is almost the same as above and will ensure that you get the newest version(s).

[5]:

potential = am.load_lammps_potential(potid='2004--Zhou-X-W-Johnson-R-A-Wadley-H-N-G--Ni')
print(potential)

2.1.3. By categoryÂ¶

The LAMMPS potentials can also be (partially) explored by using the pair_style, elements, and symbols parameters. For elements vs.Â symbols

  • elements are the specific elements that potentials for modeling atoms are representing. A model may not be associated with an element if it is a meta or coarse-grain model.

  • symbols are the model names for each unique interaction. For atomic models, this is usually the same as elements but can differ if multiple models for the same element are part of the same potential.

# Select an eam-style potential for Ni
potential = am.load_lammps_potential(pair_style=['eam', 'eam/alloy', 'eam/fs'], elements='Ni')

2.2. Using Database classÂ¶

Added version 1.3.3

More involved searches can be performed using the atomman.library.Database class. This class inherits from potentials.Database and has all of the methods for exploring the potentials and LAMMPS potentials entries as the parent class.

See the documentation Notebooks for the potentials package for more details and options.

3.1. Potential identifiers and database metadataÂ¶

Each LAMMPS potential is identified according to a potential model, i.e.Â the concept of the potential, and the implementation, i.e.Â the version of the parameter file(s). The potential is further characterized with a status that allows for older, less preferred and erroneous implementations to be filtered out.

  • potkey (str) uuid4 hash-key for the potential model, i.e.Â the published mathematical form. This is permanent once assigned.

  • potid (str) human-readable unique identifier for the potential model derived from year, author, element model information. This ideally is permanent, but may change based on updated publication information.

  • key (str) uuid hash-key for the LAMMPS potential implementation, i.e.Â the specific potential parameter file(s) and/or LAMMPS pair_style/pair_coeff command lines used. This is permament once assigned.

  • id (str) human-readable identifier for the LAMMPS potential implementation derived from the potid, source and version. This may change if potid changes.

  • status (str) indicates the current status of the potential implementation

    • âactiveâ = any implementation that is not known to be incorrect or superceded by a newer version.

    • âsupercededâ = a correctly implemented version that has been replaced by a newer version to address minor issues.

    • âretractedâ = an incorrectly implemented version due to either critical errors with the parameterization, file format, or metadata in the associated potential_LAMMPS record.

3.3 Atom-type propertiesÂ¶

Each unique atomic model is associated with a symbol, and that symbol can then be associated with an element, a mass and/or a charge.

  • symbols (list) The symbols used to uniquely identify the different atomic models defined by the potential.

  • elements() (list) Returns the list of elements that correspond to the symbols. If a symbol is not associated with an element, then that symbolâs element is given as None.

3.4.1. Potential directoryÂ¶

For potentials with parameter files, LAMMPS needs to know where the files are in order to read them in. With the Potential class, this is handled with the pot_dir parameter

  • pot_dir (str or None) the directory containing files associated with the potential. If None, then the files are assumed to either be in the working directory or a directory that LAMMPS knows about.

[18]:

Depending on how the potential was loaded, the default value of pot_dir may be different:

  • If load_lammps_potential() is used with getfiles=True, then the parameter files will be copied to the current working directory in a subfolder named for the potentialâs id. pot_dir will then match the potentialâs id. This is useful for packaging complete calculations and if you are using potentials from both local and remote locations.

  • If load_lammps_potential() is used with getfiles=False and the potential is retrieved from the local database copy, then pot_dir will be set to the full path to the library directory where the potentialâs parameter files are stored. This is convenient for purely local users as it allows for the potentials to work properly without copying the files.

  • If load_lammps_potential() is used with getfiles=False the the potential is retrieved from the remote database, then pot_dir will be set to None. This is done as the method will have no information about where the parameter files would be located locally, if they even are.

  • If a potential is taken from the loaded potentials of a Database object, then pot_dir is None.

  • If a potential is loaded from a file using atomman.lammps.Potential, the default pot_dir value is None, but it can be set during initialization using the pot_dir parameter.

5. potential_LAMMPS Data ModelÂ¶

This section describes the structure of the potential_LAMMPS data model and provides some examples for different LAMMPS pair styles.

5.1. potential_LAMMPS data model schemaÂ¶

The potential_LAMMPS data model is consistent with a json/xml structure. The root element of the model is âpotential-LAMMPSâ.

Elements:

  • âkeyâ : a uuid hash-key that uniquely identifies the potential implementation. Corresponds to the Potential.key attribute.

  • âidâ : a human-readable identifier that uniquely identifies the potential implementation. Corresponds to the Potential.id attribute.

  • âpotentialâ : contains metadata associated with the potential model.

  • âunitsâ : the LAMMPS units option to use with the potential. Corresponds to the Potential.units attribute.

  • âatom_styleâ : the LAMMPS atom_style option to use with the potential. Corresponds to the Potential.atom_style attribute.

  • âallsymbolsâ : boolean flag that indicates if all of a potentialâs symbols need to be listed in the pair_coeff lines. This is required for certain pair_styles. Assumed to be False if field is missing.

  • âstatusâ : label characterizing the known status of the potential, i.e. âactiveâ, âsupercededâ, or âretractedâ. This makes it possible to separate old/bad implementations from current ones. If field is missing, status is assumed to be âactiveâ.

  • âatomâ : provides information relating to the atom-models defined in the potential.

  • âpair_styleâ : classifies the terms that appear in the LAMMPS pair_style command associated with the potential.

  • âpair_coeffâ : classifies the terms that appear in the LAMMPS pair_coeff command associated with the potential.

  • âcommandâ : classifies the terms that appear in any other LAMMPS command associated with the potential.

5.1.1. The âpotentialâ branchÂ¶

The âpotentialâ branch gives metadata associated with the potential model that the LAMMPS implementation is based on. The potential model is the concept (mathematical expression) of a potential, and multiple implementations of the same potential model may exist.

Subelements:

  • âkeyâ : a uuid hash-key that uniquely identifies the potential model. Corresponds to the Potential.potkey attribute.

  • âidâ : a human-readable identifier that uniquely identifies the potential model. Corresponds to the Potential.potid attribute.
